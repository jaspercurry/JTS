# W3.1 — Single entry + active/passive declaration (Codex prompt)

Read [`README.md`](README.md) (the charter) first; it is binding. Then
read this file completely before writing code.

## Context

`/correction/` is the entry point every setup path funnels into — the
landing page's "Room correction" tile, the plain-HTTP preflight page's
default Proceed button, and nginx's `/correction/proceed` no-JS
fallback all resolve to it. Today it does nothing of its own: the
`do_GET` ladder in `jasper/web/correction_setup.py` treats `"/"` and
`"/room"` as the same route —

```python
if path in {"/", "/room"}:
    ctx = begin_request(self)
    self._send_html(_render_page(
        cfg["hostname"], ctx["csrf_token"], ctx["flash"],
    ))
    return
```

— so `/correction/` silently *is* the Room-correction page, with no
acknowledgment that a household with an active crossover needs to set
that up first. `campaign/punchlist.md` calls this out explicitly in
its "UX-architecture facts for the rethink" note (after item #38):
`/correction/` root = alias of `/room/` + preflight default (not dead
code; vestigial hub concept)`.

The active/passive fact itself is currently **derived, never
declared**: `jasper/active_speaker/setup_status.py`'s
`read_active_speaker_setup_status()` computes `active_group_count`
from the saved `OutputTopology` (`_active_group_count`, line 115) and
short-circuits to `ROOM_AUTHORITY_PASSIVE_NOT_REQUIRED` when it is
zero (lines ~253–262). This is correct and must stay correct — it is
the actual safety authority Room's own page independently re-checks
on every load via `_room_correction_readiness()`
(`jasper/web/correction_setup.py:2116-2131`) before rendering anything
that could touch playback. What is missing is a **household-facing
first step**: nothing today asks "does this speaker use an active
crossover?" up front, so a household with passive/simple speakers has
no reason to expect a "Crossover" tab, and a household setting up an
active speaker can wander into `/correction/room/` and only discover
it is blocked after clicking in.

Evidence for the resolved IA decision (do not re-litigate): Jasper's
own words, recorded verbatim in
`campaign/OVERNIGHT-2026-07-16-NIGHT2.md` under "Board at time of
writing": *"active/passive declaration = step one (stored durable
fact, wizard-owned; derived topology check stays the safety
authority); room tab gated on declaration"*.

## Requirements

### The declaration record (`crossover_declaration.py`)

Mirror `household_mic.py` exactly in shape:
- `DEFAULT_CROSSOVER_DECLARATION_PATH = Path("/var/lib/jasper/correction/crossover_declaration.json")`
  — same directory as `household_mic.json`, same file-mode convention
  used elsewhere in `jasper/correction/` (0644 — this is a UX fact,
  not a secret; match whatever mode `household_mic.py` uses).
- A frozen dataclass, e.g. `CrossoverDeclaration(declared_active: bool,
  declared_at: float, schema_version: int)`. Keep it this small — no
  speculative fields (no "declared_by", no free-text notes; the
  household-mic record's richer shape exists because it tracks
  calibration provenance, which this fact has no equivalent of).
- `load_crossover_declaration(*, path=...) -> CrossoverDeclaration |
  None`, `store_crossover_declaration(declared_active: bool, *,
  path=...) -> CrossoverDeclaration`, `clear_crossover_declaration(*,
  path=...) -> None`. A missing or malformed file reads as `None`
  (undeclared) — never raises, never guesses.
- Module docstring states explicitly, in the first paragraph: this
  record is **UX gating only**; the safety authority for whether Room
  correction may run is `jasper.active_speaker.setup_status.
  read_active_speaker_setup_status`, which this record does not feed
  and must never feed.

### The entry screen (`correction_entry_flow.py` + its JS)

Three states, server-driven (the page renders what the backend says;
it does not compute eligibility client-side — same discipline as
every other correction page):

1. **Undeclared** (`load_crossover_declaration()` returns `None`):
   ask the one question — "Does this speaker use an active
   crossover?" — with plain-language help text (a sentence, not a
   glossary entry: what an active crossover is, phrased for someone
   who has never heard the term). Two actions, both POSTs to your new
   declare endpoint: declare passive, declare active. No third
   option, no skip.
2. **Declared passive**: call `read_active_speaker_setup_status()`
   (fresh, the same call `_room_correction_readiness()` already makes
   — reuse `_room_correction_readiness()` itself if its signature fits,
   rather than re-deriving the call). If `active` is `False` (topology
   agrees), show a short confirmation and a single next action to
   `/correction/room/`. If `active` is `True` (topology disagrees —
   the **named refusal** case), do not silently trust the declaration:
   show a named message stating the contradiction plainly ("Your
   speaker's output setup is configured with an active crossover, but
   you declared it passive.") with two actions — re-declare as active,
   or go fix speaker setup (`acoustic_commissioning.setup_href` from
   the same status read). Log this contradiction at WARN with a stable
   `event=` line (see Observability below) — it should never happen in
   normal use, so it is worth knowing about if it does.
3. **Declared active**: read the same status. If Room is allowed
   (`room_correction_allowed` true), same as the passive-agreeing case
   but naming that active-crossover setup is done. If not yet ready,
   show the existing `acoustic_commissioning.detail`/`setup_href`
   (do not write new copy for this state — reuse Active's own message,
   the same way Room's blocker card already does) with the single next
   action pointed at `/correction/crossover/`.

A **re-declare** control is available in every state (small,
secondary, not competing with the primary action) — a household that
changes its speaker hardware must be able to correct a stale
declaration without editing a file by hand.

### Room-tab gating (`correction_hub.py`)

`section_tabs()` gains one new parameter carrying the Room-tab lock
state — pass through the same `room_correction_allowed`/`setup_href`
pair the entry screen already computed (don't have `section_tabs`
call `read_active_speaker_setup_status()` itself; that would be a
second call site for the same read on every page load across room,
crossover, and bass — thread the value through from each caller,
which already has it or can cheaply get it).

- When Room is allowed: render the "Room" tab exactly as today (a
  clickable `<a>`).
- When Room is not allowed: render it as a non-link element (a `<span>`
  or `<button disabled>`, matching whatever the canonical design
  system's disabled-affordance pattern is elsewhere — check
  `app.css` for an existing disabled/locked treatment before inventing
  one) with a lock icon or equivalent visual cue and a `title`/
  `aria-label` naming why (reuse the same detail string, don't write a
  second one).

This is a **preview affordance only** — `/correction/room/` itself
must keep independently gating via `_room_correction_readiness()`
exactly as it does today (do not remove or weaken that check; a direct
URL visit, a stale tab render, or a race between declaration and
topology must still hit the real gate).

### Observability

One `event=correction_entry.declared` line (via `jasper.log_event`) on
every successful declare-POST, carrying `declared_active` and whether
it matched or contradicted the topology-derived `active` value at that
moment. One `event=correction_entry.declaration_mismatch` WARN line
specifically for the named-refusal case in requirement 2 above (kept
separate from the general declare event so it is easy to grep for in
isolation — "how often does the household's own claim disagree with
reality" is a useful metric on its own).

## Anti-overengineering fences

Do NOT: add a generic "wizard step" framework or a reusable "gated
tab" component beyond the one parameter `section_tabs()` gains; store
anything about *why* the household chose active/passive (no free-text
reason field); add a second safety check that duplicates
`read_active_speaker_setup_status`'s derivation in Python or JS; make
the declaration record readable/writable from more than the one module
you create (every other surface that needs the fact calls
`load_crossover_declaration`, never reads the JSON file directly); add
websockets/SSE to the entry page (poll like the sibling pages do, if
polling is needed at all — a single-question static page may not need
to poll faster than once, on load); touch `nginx-jasper.conf` routing
behavior (comments only, and only if genuinely needed for clarity).
