## Requirements

### Crossover envelope copy (jargon)

1. Both `"...their exact repeat ledger is not complete..."` verdict
   strings — rewrite to name the user's actual situation without the
   internal term "ledger". These verdicts fire when a driver's
   near-field acoustics are saved but a repeat-completeness check
   isn't satisfied for the current topology/profile; state that in
   plain terms ("Run the driver measurement again to finish the
   repeat check" style — match the Language guide's imperative,
   task-first phrasing, e.g. "Measure {role} again").
2. `"Repeat safety state is unavailable. Run the microphone level
   check again before another sweep."` — same treatment; "repeat
   safety state" is internal-mechanism language for "JTS lost track of
   whether this measurement is still safely repeatable."

### `safe_cap_below_window` advice duplication

`ramp.py`'s raw error currently embeds advice
(*"...raise the external amplifier and retry"*) that
`level_match.py`'s mapped user-facing copy repeats independently
(*"Raise the external amplifier a little, then retry."*) — two
independently-worded copies of the same instruction, one of which
(`ramp.py`'s) is meant to be an internal fact string, not
user-facing copy (its consumers include log lines and the `error`
field other code branches on, not just the mapped-copy path). Trim
`ramp.py`'s raw string to the **fact only** (the cap was reached below
the target window — no imperative verb, no "raise the amplifier"
clause); keep `level_match.py`'s user-facing copy exactly as the one
place that gives advice. Confirm nothing else pattern-matches on the
advice clause specifically (the `_RAMP_REFUSAL_PREFIX_COPY` matching
is prefix-based on the *fact* clause, not the advice clause — check
this holds before trimming).

### Capture-page copy (run-18/21 findings)

1. **Mic-label consistency**: pick one of "Automatic / phone default"
   / "Automatic (recommended)" as the single canonical label for "no
   explicit device chosen" across both picker instances — "Automatic
   (recommended)" is the more actionable/confident phrasing (it tells
   the household what to do, not just what happens by default); use
   it in both places unless your preflight reading finds a reason the
   two pickers need different wording (e.g. one truly has a different
   default behavior) — if so, name that reason in the PR description
   rather than silently picking one.
2. **USB ids in mic labels**: the browser-provided `MediaDeviceInfo.label`
   sometimes includes a raw vendor/product identifier
   (`something (xxxx:xxxx)`-shaped suffixes are a known
   `getUserMedia` behavior on some Android/Chrome + USB-audio-class
   device combinations). Add a display-only cleanup (strip a trailing
   parenthesized hex-id-shaped suffix) purely for the rendered option
   text — never for the value submitted to the Pi, which must keep
   using the exact browser-reported label/deviceId the calibration
   device-mismatch backstop
   (`correction_setup._calibration_device_mismatch`) already depends
   on for its heuristic. Keep this narrow: a regex for the specific
   `(xxxx:xxxx)`-shaped suffix, not a general label-rewriting engine.
3. **"put this phone down" contradiction**: the level-match-locked
   note (*"Level matched. The speaker will continue on its own — you
   can put this phone down."*) fires once a driver's level lock
   completes. Trace the multi-driver sequential flow (woofer lock →
   tweeter lock in the same relay session, per the v3 session-spanning
   protocol — see `README.md` "State of the world") and confirm
   whether this note can render immediately before the SAME session
   needs the phone again for the next driver's lock. If it can, the
   note is dishonest at that point in the flow — either suppress it
   when another lock is about to start in the same session, or reword
   it to say "you can set the phone down for a moment" only when the
   flow genuinely has nothing further queued.
4. **Tweeter-lock calibration re-prompt after one-tap confirm**: trace
   whether a household that used the stored-calibration one-tap
   confirm (`#1564`, `mode: "stored"`) for the FIRST driver in a
   session gets prompted with the raw "Choose a calibration file."
   default (upload-mode) UI for a SUBSEQUENT driver in the same
   session, instead of the calibration choice persisting across
   drivers within one relay session. If confirmed, fix it so the
   chosen calibration mode/identity persists for the duration of the
   session (mirror however device selection already persists via
   `DEVICE_STORAGE_KEY`/`storedDeviceId()` — same pattern, scoped to
   calibration choice).
5. **No user-visible page build marker**: `version.json` is fetched
   for protocol-compat validation only. Add a small, unobtrusive
   footer/corner marker (e.g. "build {short version}") rendered from
   the same already-fetched identity object — no new network request,
   no new endpoint. Keep it visually minor (this is a diagnostic aid
   for pilots/support, not a feature); do not add a settings page or
   any interactive element around it.

## Anti-overengineering fences

library for any of the above — each fix is a small, targeted change at
its named site; add a copy-lint/glossary-enforcement test suite beyond

linter is out of scope — this is a manual sweep); change the

persistence — this fix is about within-session continuity only, don't
duplicate the cross-session mechanism).
