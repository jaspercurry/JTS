## Context

The active-speaker driver layout, LLM research-prompt/import step,
manual crossover settings, staging preview, and the commissioning
RESET control all live on `/sound/` today, tucked inside a `<details
class="advanced" data-active-speaker-setup>` disclosure titled
"Speaker setup" (`deploy/assets/sound-profile/js/main.js`,
`renderActiveSpeakerSetup()` around line 716, mounting
`renderOutputTopologySetup()` around line 1879 — this second function
and everything it calls is the whole surface being moved). A bonded
active follower mounts the **same** function tree expanded as its
page's *primary* content instead of behind a disclosure
(`renderFollower()`, ~line 531) — see the docstring on
`_follower_sound_html` in `jasper/web/sound_setup.py` (~line 1537):
*"the local crossover ... stays with the speaker that owns the DAC
path"* is a load-bearing product promise this workstream must keep
true, just from a different URL.

Meanwhile `/correction/crossover/` (the measurement flow) already
reads the SAME durable design draft
(`/var/lib/jasper/active_speaker_design_draft.json`, owned by
`jasper/active_speaker/design_draft.py`) that `/sound/`'s section
writes, and its envelope offers `edit_manual` actions
(`jasper/active_speaker/crossover_envelope.py`, four call sites —
search `"id": "edit_manual"`) whose `href` is a bare `"/sound/"` —
today, editing the crossover mid-measurement-flow means leaving the
flow entirely. Jasper's own words (`campaign/OVERNIGHT-2026-07-16-NIGHT2.md`,
2026-07-16 evening directive #5): *"reset button migrates from /sound/
into the flow (W3.2)"* — the RESET control specifically, because
resettability is itself under test as part of the from-scratch E2E
milestone (see `README.md`).

**This is a page-surface move, not a state migration.** The design
draft file, its schema, and every function that reads/writes it stay
exactly where they are — only *where a household clicks to edit it*
changes.

## The staged-PR plan (this workstream is multiple PRs by design)

Per the charter's "one PR per workstream unless the prompt splits it"

(acceptance commands → adversarial review clean → green CI →

### Stage A — shared-module extraction (no user-visible change)

Extract, do not yet re-home:

- **JS**: move `renderOutputTopologySetup` and its full call subtree
  out of `deploy/assets/sound-profile/js/main.js` into a new shared
  module (propose an exact path under `deploy/assets/shared/js/` or a
  new `deploy/assets/correction/js/crossover/active-speaker/` — pick

  screenshot or DOM diff on `/sound/`.
- **Python**: extract the `/active-speaker/*` handler *bodies* (not
  the HTTP dispatch lines — the functions those lines call) from
  `jasper/web/sound_setup.py` into a new shared module,
  `jasper/web/active_speaker_flow.py`. `sound_setup.py`'s `do_GET`/
  `do_POST` ladder keeps its exact same routes, now calling into the
  shared module — again zero behavior change, verified by the existing
  `/sound/` active-speaker test suite passing unmodified (find it —
  likely `tests/test_web_sound_setup*.py` filtered to active-speaker
  cases; confirm the exact file during preflight).

File allowlist for Stage A: the two new files above, plus additive

`jasper/web/sound_setup.py` (replace, don't duplicate — the old code

## Invariants (apply to every stage)

- **SSOT**: the design draft file and `jasper/active_speaker/design_draft.py`
  are the only reader/writer at every stage. No stage introduces a
  second copy, cache, or client-side mirror of its contents beyond
  what already exists for rendering.
- **Follower parity**: at every stage, run the follower-mode check
  (`_follower_sound_html`'s docstring's own promise) on a bonded
  follower JTS3 config, not just a solo one — this is the single
  easiest invariant to silently break in Stage C.
- **No mixed content**: every new HTTPS-side route in Stage B is
  actually reachable over HTTPS through the existing `/correction/`
  nginx proxy — verify with `curl -k https://jts3.local/correction/crossover/active-speaker/...`
  (or your chosen path), not just against the Python test client.
- **RESET stays a real reset**: `_reset_output_topology_payload`'s
  existing behavior (stop any active tone/summed-test/safe-playback
  session, reset to detected passive, clear active-speaker setup
  state) is unchanged by the move — Stage A's extraction test should
  assert byte-identical behavior before/after, and Stage B's new route
  should call the exact same function, not a re-derived version.

## Anti-overengineering fences

Do NOT: build a generic "shared wizard section" framework beyond the
one module this workstream extracts; add a second design-draft
writer, cache, or "draft-in-progress" client state; keep the old
`/sound/` inline-render code path alive "as a fallback" past Stage C
(delete it in Stage D, don't leave two live implementations); add a
migration/versioning concept for the design-draft schema (it isn't
changing); invent a new page-shell abstraction — the crossover page's
new phase mounts the shared module the same way `/sound/` did, just
from a different page shell.
