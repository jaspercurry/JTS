## Context

`campaign/VALIDATION-FIX-PLAN.md`'s Wave 3 framing and
`campaign/OVERNIGHT-2026-07-16-NIGHT2.md`'s scope note both name this
as an architectural question, not a fix: **alignment and phase
decisions near a crossover point only need good SNR/quality evidence
in the region around that crossover — they do not need the driver's
entire nominal passband to be pristine.** Whether today's pipeline
actually over-constrains those decisions by gating on full-passband
quality, and if so what the right architectural fix is, is what this
workstream investigates.

The immediate SYMPTOM that motivated this question is already fixed
and is **not** what you are re-diagnosing: punchlist #51 (the
campaign's largest finding, `campaign/punchlist.md`, "FORENSICS
2026-07-17") found that `_capture_to_magnitude`
(`jasper/active_speaker/driver_acoustics.py`, ~lines 510–636 at the
time of that investigation) deconvolved quiet-window noise through the
sweep's own matched filter and a fixed 505 ms arrival window, which a
constant-epsilon Tikhonov regularization amplified at the sweep's 5 ms
fade-in edge — overstating sub-bass noise by 40–50 dB. That estimator
bug is fixed by #1563 (`a9dad24bc`, "per-band noise estimate must
match measured reality"), verified on hardware: true worst-band SNR at
normal desk placement reads 60+ dB, not the false 11–16 dB the
pre-#1563 pipeline reported. Ground truth was always fine; the
estimator lied.

The **architectural** question survives the estimator fix: even with
a trustworthy per-band SNR number, is the pipeline asking the right
question at the right scope? Two pieces of evidence, from reading the
current code (verify both during your own investigation — do not
trust this prompt's summary without re-checking):

- `jasper/active_speaker/driver_acoustics.py`'s `analyze_driver_capture`
  (docstring, ~line 965) computes its admission gate over
  `relevant_hz = passband_hz ∩ [ANALYSIS_LO_HZ, ANALYSIS_HI_HZ]`, where
  `passband_hz` comes from
  `jasper.active_speaker.commissioning_capture.driver_passband_hz`: for
  an "open end" (a woofer's low side, a tweeter's high side), that
  function clamps to the full trusted analysis window
  (`ANALYSIS_LO_HZ`=40 Hz / `ANALYSIS_HI_HZ`=18000 Hz), not to
  anything scoped near the crossover. A woofer whose only crossover is
  at, say, 300 Hz gets an admission gate spanning 40 Hz–300 Hz — its
  entire nominal passband, including deep bass content nowhere near
  where any alignment/phase decision actually happens.
- The SAME file's `_overlap_band_levels`/`OVERLAP_BAND_RATIO` (~lines
  813–880, `OVERLAP_BAND_RATIO = 2**0.5`, "half-octave each side → one
  octave total" around each `fc`) already computes a genuinely
  crossover-region-scoped SNR verdict — used specifically for the
  overlap-band polarity/delay decision at the driver's crossover
  point. This narrow, region-scoped check already exists and already
  works; it is not what admits/rejects a driver's capture as usable in
  the first place.

So there appear to be two different gates in the same file, one
region-scoped (overlap-band, used for alignment-adjacent decisions)
and one full-passband (admission/usability, used to decide whether the
capture is trustworthy at all) — and the run-19/#47/#49 saga (a driver
repeatedly refused/insufficient over several hardware runs before the
estimator fix) may be evidence that the full-passband gate is stricter
than the actual downstream use requires, independent of whether the
estimator was lying. **Confirm or refute this framing yourself** —
it is this prompt's hypothesis, not an established fact.

Separately, `jasper/audio_measurement/null_walk.py` (the bounded
null-walk that resolves polarity via null depth and delay via a
manual path-difference seed — see AGENTS.md's active-crossover
section) does not appear to consume `band_snr_verdicts`/
`worst_band_verdict` at all; it works directly from captured null
depth. Confirm whether alignment/phase work is actually gated by the
full-passband admission check upstream (i.e., a driver capture must
pass the full-passband gate before `null_walk` ever runs on it) or
whether the two are more decoupled than that — this matters a great
deal for how narrow or broad the eventual design fix needs to be.

## Investigation method

1. **Map the actual gate graph.** For every place a driver capture's
   usability affects a downstream decision (admission into the design
   draft, feeding the level solver, feeding null-walk alignment,
   feeding the summed-crossover blend check), trace which SNR/quality
   computation gates it and over what frequency range. Produce a table
   (driver decision → gating function → frequency scope → why that
   scope was chosen, if discoverable from code comments/docs/git
   history).
2. **Test the hypothesis against the run-19/20 evidence.** Using the
   ground-truth numbers punchlist #51 recovered from the pulled
   hardware WAVs (raw sub-bass ambient ≈ −75 dBFS, true worst-band SNR
   ≈ 64–66 dB), determine: would a region-scoped admission gate
   (crossover-region only) have accepted the SAME drivers faster or
   differently than the full-passband gate did, holding the estimator
   fix constant? If the full-passband gate would have admitted the
   driver just as fast once the estimator was honest, the architectural
   concern is weaker than hypothesized — say so plainly; a null result
   here is a valid and useful finding.
3. **Identify what actually needs region fidelity vs what genuinely
   needs full-passband quality.** Some decisions plausibly DO need
   full-passband SNR (e.g., the level solver's own SNR-floor logic,
   which is about whether the WHOLE measured curve is trustworthy
   enough to compute a level from — check whether that's actually true
   or whether it too only needs specific bands). Alignment/polarity/
   delay decisions plausibly only need the crossover-region window
   `_overlap_band_levels` already computes. Name each decision
   explicitly; do not assume "narrower is always better" — a
   region-scoped gate that lets through a genuinely bad capture into a
   decision that silently assumed full-band trustworthiness would be a
   regression, not an improvement.
4. **Consider the "no change needed" outcome seriously.** If the
   investigation concludes the current full-passband admission gate is
   already the right scope (e.g., because a driver's FULL passband
   quality is legitimately relevant to whether its captured curve can
   be trusted for shaping decisions elsewhere, not just alignment),
   that is an acceptable, valuable conclusion — document why, and this
   workstream's deliverable becomes "confirmed correct, no
   architectural change" rather than a redesign.

## Deliverable and gate

**Primary deliverable: a design note**, added as a new file
`docs/research/2026-07-1X-region-scoped-quality-gates/README.md`
(pick the actual date; follow the shape of the existing
`docs/research/2026-06-19-active-crossover-calibration/` directory as
a structural example — a dated research artifact, not a HANDOFF). It
must contain:

- The gate-graph table from investigation step 1.
- The run-19/20 hypothesis test result from step 2, with numbers.
- A named recommendation for each decision from step 3: keep
  full-passband, narrow to crossover-region, or "needs Jasper's
  judgment call — here are the tradeoffs" (this workstream does not
  have standing authority to change a safety-adjacent quality
  threshold; a genuine design fork goes to Jasper as an open decision,
  not a coin flip).
- **A hardware test plan for validating any recommended change**,
  explicitly needing Jasper's own physical measurements: if the
  recommendation narrows any gate to a crossover-region window, the
  validation plan must specify what physical measurement (e.g.,
  driver-to-driver path-length/mm offset at the actual cabinet
  geometry, feeding `null_walk`'s manual path-difference seed) Jasper
  needs to supply for a hardware pass to distinguish "the narrower
  gate correctly admitted a driver that alignment can still safely use"
  from "the narrower gate let through a capture that produces a wrong
  alignment result." Be concrete about what to measure, not just that
  measurement is needed.

**A small, narrowly-scoped proof-of-concept implementation is
authorized ONLY if** your investigation produces a clear, low-risk
recommendation AND the PoC is read-only/observational (e.g., computing
what a region-scoped verdict WOULD have said, alongside the existing
gate, logged for comparison — never replacing or weakening the
existing gate's actual admission decision). If you build this
PoC, it stays inside the design-note deliverable (a script or a small
diff described and linked from the README, not a merged behavior
change) and does not need the full W3.1–W3.6-style adversarial
code-review gate — it needs the design note itself reviewed.

**What does NOT ship from this workstream**: any change to
`analyze_driver_capture`'s actual admission logic, any change to
`QualityModel` thresholds, any change to what gates `null_walk`. Those
are follow-up implementation work for a future, separately-scoped
wave, once Jasper has read the design note and made the judgment calls
it surfaces.
