## Context

The crossover envelope conflates two different kinds of "done" into
one flat stepper array. `_STEP_IDS = ("speaker_setup", "microphone",
"drivers", "alignment", "apply")` (`jasper/active_speaker/
crossover_envelope.py:29`) and the single `done: set[str]` built up
through `build_crossover_envelope` (~lines 734–755) mixes:

- **run-journey facts** — has *this* measurement attempt gotten
  through the microphone level check, the driver sweeps, alignment —
  which reset to nothing at the start of a fresh attempt, and
- **durable-achievement facts** — is there *currently* a working,
  applied crossover on the speaker, which persists across attempts and
  says nothing about whether a NEW attempt is in progress.

Concretely: `done.add("apply")` (line 753) fires whenever
`applied_ready and not automatic_remeasure and not
strict_isolated_complete` — a durable fact about the speaker's current
state, unrelated to the run-journey steps before it. Separately,
`done.discard("microphone")` (lines 818, 884) retroactively removes a
run-journey step from an already-built `done` set when a controller
target is blocked or orphaned. The `done_manual` screen (line 1073)
hits exactly this combination on hardware — punchlist #36 (a
2026-07-16 finding): *"manual-applied keeps apply∈done while
interrupted retune discards microphone → 1✓, 2-4 pending, 5✓;
done_manual presents as clean terminal, hides the interrupted
attempt."* A stepper that can render step 1 and step 5 done with
steps 2–4 pending, on a screen whose own verdict text calls itself
"applied and ready," is not a stepper — it's two unrelated facts drawn
with one widget.

The page shell compounds this: `jasper/web/correction_crossover_flow.py`'s
static header —

```html
<p class="eyebrow">Speaker layer</p>
<h2 class="section__title">Calibrate the active crossover</h2>
<p id="crossover-verdict" class="form-hint">Checking the speaker…</p>
```

— renders once and then has its `#crossover-verdict` text replaced by
whatever the envelope's `verdict_text` says on every poll, for every
screen from "declare a driver layout" through "applied and ready."
The card's own copy ("Calibrate the active crossover") is itself a
durable-achievement claim, but it sits above a paragraph that is
constantly being overwritten with run-journey narration — the two
kinds of information are visually stacked as one card with no seam
between them.

## Requirements

### Split durable achievement from run-journey

`_STEP_IDS` shrinks to the genuine run-journey:
`("speaker_setup", "microphone", "drivers", "alignment")` — four
steps, each meaning "has THIS attempt gotten through this step,"
reset to not-done whenever a fresh attempt/retune begins (this is
already partially true via the `done.discard` calls; keep that
discard behavior, just stop `done` from ever containing `"apply"` as
a fifth stepper entry).

"Apply" becomes a **separate status card**, not a stepper step: does
the speaker currently have a working applied crossover
(`applied_ready`/`automatic_applied`/`tuning_owner`), rendered
independently of wherever the run-journey stepper currently sits. This
status card is what the static "Speaker layer / Calibrate the active
crossover" header becomes — pure status ("Applied: manual crossover,
ready for room correction" / "Not yet applied" / "Interrupted — a
previous attempt did not finish"), never overwritten with in-progress
run narration. Add a new envelope field for this (e.g.
`applied_status: {applied: bool, owner: str|None, detail: str}` —
propose the exact shape, keep it small) rather than inferring it
client-side from `steps`/`screen`; the server-driven discipline every
other correction page follows applies here too.

The `done_manual` reproduction case (or its current-code equivalent,
per the preflight note above) must render as: applied-status card says
"applied, ready" (true — a manual crossover IS currently applied);
run-journey stepper shows an HONEST state for the interrupted attempt
(the existing `crossover_done_manual_retune_interrupted` nudge already
names this — keep it, and make sure the stepper it sits above agrees
with it instead of contradicting it with a stray step-5 checkmark).

### Instructions render only right before their action

Audit every static instruction string currently rendered on
`/correction/crossover/` (Pi-side page) that describes an action the
**phone** performs (opening the phone capture link, holding the phone
steady, tapping Start, waiting for a tone). Any such instruction that
is visible on the Pi page for more than the single screen immediately
preceding that phone action is a violation of rubric point 7. Move
that narration onto the phone page itself (`capture-page/`) if it
isn't already there, and reduce the Pi-side copy to what a person
standing at the Pi actually needs at that moment (usually: "waiting
for the phone" plus a Stop control, not a re-statement of what the
phone should be doing). Punchlist #46's *"'put this phone down' copy
lies"* finding is a concrete instance — trace where that string lives,
confirm on which screens it renders, and fix the step-scoping (it
should describe an action true of exactly the step it's attached to,
not linger into steps where the phone is doing something else or
sitting idle already).

## Acceptance commands

```
.venv/bin/pytest tests/test_active_speaker_crossover_envelope.py \
  tests/test_web_correction_*.py -q
bash scripts/check-js-syntax.sh
node tests/js/<your-new-harness>.mjs
scripts/test-fast
```

**Your new `tests/js/*.mjs` harness is local-only — you MUST run it
by hand before every push.** CI's `js` job
(`.github/workflows/tests.yml`) invokes a *curated, hardcoded* list of
harnesses; a newly-added harness is NOT in that list and will NOT run
in CI, and the charter forbids editing `.github/`.
`scripts/check-js-syntax.sh` only `node --check`s syntax, it does not
execute the harness. So the explicit `node tests/js/…` line above is
the only thing that actually exercises your new test — green CI does
not prove it passes.
