# W3.4 — Room flow parity (Codex prompt)

Read [`README.md`](README.md) (the charter) first; it is binding. Then
read this file completely before writing code. Prereq: W3.3 merged —
this workstream applies the same standards to the room flow, and
should read as "does Room already meet the bar W3.1–W3.3 set for
Crossover, and if not, fix the gap" rather than a mechanical port.

## Context

Room correction (`jasper/correction/envelope.py` +
`jasper/correction/session.py`, rendered by
`jasper/web/correction_setup.py`'s `_render_page`) is architecturally
different from the crossover flow it is being brought to parity
with — it is not a discrete stepper. It uses a server-owned
**whole-page section visibility** model
(`_sections_for`/`_SCREEN_SECTIONS` in `envelope.py`) plus a scalar
`_progress(screen)` (`position`/`total`/`labels` from
`_PROGRESS_SPINE`), documented as the design of record in
`docs/room-correction-information-design.md`. Do not assume Room has
the exact same "durable achievement mixed into a stepper array" bug
W3.3 fixed on the crossover side — its data model is different enough
that the bug may not translate literally. Your job is to **check each
of the three standards below against Room's actual current code**,
fix any real gap you find, and report "already met, no change needed"
for any standard Room already satisfies — do not invent a fix for a
problem that isn't there.

Refusal-naming for Room's AGC case specifically was already fixed in
Wave 1 (W1.1, punchlist #31/#32): `_level_match_refusal_failure`
(`jasper/correction/envelope.py:370`) routes the level-match ramp's
terminal code through `jasper.correction.level_match.
describe_ramp_refusal` into the envelope's `code`/`text`/`retryable`/
`recovery_action` shape. This workstream is not re-doing that fix — it
is auditing whether that pattern's coverage is **complete** across
every other refusal-shaped terminal Room can reach.

## Requirements — the three standards, audited then fixed

### 1. Refusal-naming completeness

Walk every terminal/failure path `build_envelope` can render (every
branch that sets a `failure`/`verdict` with a code, plus every place
`_nudges` surfaces a warn-severity entry) and confirm each one carries
a named code + household-actionable text, the same shape
`_level_match_refusal_failure` already produces for the AGC case. Any
path that still surfaces a bare exception class name, a raw Python
`str(exc)`, or an internal vocabulary term (see standard 3) instead of
a named code+copy pair is a gap — fix it using the SAME
`describe_ramp_refusal`-style mapping pattern (extend the existing
table; do not invent a second mapping mechanism).

This audit is scoped to **envelope-rendered household-facing
refusals** only — not the ~130 generic `ValueError`s in
`correction_setup.py` that return 400s for malformed API requests
(wrong content-type, missing required field, unknown route). If you
are unsure whether a given raise site is household-facing, trace
whether its message can reach `build_envelope`'s output; if it only
ever reaches a raw HTTP 400 body that the browser's own request never
surfaces to the user (a client-code bug path, not a user-reachable
state), it is out of scope.

### 2. Step-scoped instructions

Using the same method W3.3 applied to the crossover page: find every
static instruction string on `/correction/room/` that describes a
phone-side action, and confirm it renders only on the section/screen
immediately preceding that action — not persistently across
unrelated sections. Room's `_sections_for` already computes which
sections are visible per screen (this is more granular than
crossover's old flat stepper was before W3.3), so this audit may find
Room already mostly satisfies this standard — confirm rather than
assume, and fix any genuine leftover case (e.g. a capture-setup
instruction that also renders during analysis or verification, where
the phone is not doing that action).

### 3. Copy standards (Room-specific Language-guide sweep)

`docs/active-crossover-information-design.md`'s "Language guide"
section is written for the crossover flow specifically; Room has its
own copy surface and may have its own jargon leaks that guide doesn't
cover verbatim. Grep Room's rendered strings
(`envelope.py`'s verdict/nudge text, `correction_setup.py`'s static
page template) for internal vocabulary a household member would not
recognize — session/state-machine terms, internal field names, code
identifiers — and rewrite using the same principle the crossover
Language guide states: *"the normal flow should name the user's task
and the next action"* rather than internal mechanism. List every
string you changed in the PR description with a one-line before/after
so the review can judge each individually (this is Room's equivalent
of W3.5's crossover-scoped sweep — do not wait for or depend on W3.5;
the two workstreams operate on disjoint files).

## Anti-overengineering fences

Do NOT: introduce a stepper array to Room if its current section-model
already satisfies the run-journey/durable-achievement distinction (do
not force Room's architecture to look like Crossover's just for
symmetry — different data shapes for a genuinely different UI are
fine; only fix an actual violation of the *standard*, not a difference
in *implementation shape*); build a shared cross-flow refusal-copy
component beyond reusing the existing `describe_ramp_refusal` table;
touch any measurement math, session state machine, or DSP path;
rewrite copy that already meets the Language-guide bar just for
stylistic preference.
